
# Metodologia

Ao adotar uma abordagem mais flexível, a metodologia ágil permite uma maior adaptação às mudanças e oferece maior liberdade às equipes de desenvolvimento. Isso resulta em um ambiente mais colaborativo e dinâmico, onde as equipes são incentivadas a tomar decisões de maneira autônoma.Essa estrutura não só aumenta a eficiência, mas também garante que as equipes possam realizar ajustes rapidamente, com base no feedback contínuo, levando a melhorias substanciais nos processos de desenvolvimento e, consequentemente, nos negócios. De acordo com Pressman (2010, p. 95), o Scrum é um modelo que tem se mostrado altamente eficaz na melhoria da produtividade e na qualidade dos produtos, sendo uma das metodologias mais adotadas no mundo do desenvolvimento de software.

## Controle de Versão

A ferramenta de controle de versão adotada no projeto foi o
[Git](https://git-scm.com/), sendo que o [Github](https://github.com)
foi utilizado para hospedagem do repositório.

O projeto segue a seguinte convenção para o nome de branches:

- `main`: versão estável já testada do software
- `unstable`: versão já testada do software, porém instável
- `testing`: versão em testes do software
- `dev`: versão de desenvolvimento do software

Quanto à gerência de issues, o projeto adota a seguinte convenção para
etiquetas:

- `documentation`: melhorias ou acréscimos à documentação
- `bug`: uma funcionalidade encontra-se com problemas
- `enhancement`: uma funcionalidade precisa ser melhorada
- `feature`: uma nova funcionalidade precisa ser introduzida

Discuta como a configuração do projeto foi feita na ferramenta de versionamento escolhida. Exponha como a gerência de tags, merges, commits e branchs é realizada. Discuta como a gerência de issues foi realizada.

### Processo

Backlog: A lista priorizada de funcionalidades do projeto a serem identificadas e adicionadas a qualquer momento, e que será trabalhada pelo time.
To Do: Representa a Sprint, que são as unidades de trabalho que reúnem as funcionalidades selecionadas para desenvolvimento
In Progress: São as tarefas que estão atualmente em desenvolvimento na sprint.
Done: Ocorre após a passagem pelas fases de documentação, implementação, testes e controle de qualidade, sinalizando a entrega do software ao usuário. Após essa fase, novas sprints poderão ser realizada enquanto o software continuar em uso.

Justificativa da Escolha das Ferramentas

O Visual Studio Code foi selecionado como editor de código por ser leve, possuir diversas extensões e permitir integração com o GitHub, facilitando o desenvolvimento colaborativo.

O GitHub foi escolhido para o versionamento e armazenamento do código, garantindo controle de alterações e trabalho em equipe de forma segura e organizada.

O Figma foi utilizado para a criação dos protótipos e definição do design de interface, por oferecer recursos que permitem visualizar e ajustar a experiência do usuário antes da implementação.

O kanban foi adotado como ferramenta de gerenciamento de tarefas, por possibilitar uma melhor organização do fluxo de trabalho, acompanhamento das entregas e divisão de responsabilidades dentro do projeto.

O React Native foi adotado como framework principal por permitir desenvolvimento cross-platform com uma única base de código, utilizando JavaScript e React para criar aplicações nativas tanto para iOS quanto Android. Esta escolha reduz significativamente o tempo de desenvolvimento e os custos de manutenção, além de aproveitar o conhecimento existente em tecnologias web para o desenvolvimento mobile.

O Expo foi escolhido como plataforma de desenvolvimento por oferecer um ambiente completo e simplificado para React Native, eliminando a necessidade de configuração complexa do ambiente Android/iOS. A ferramenta acelera o processo de desenvolvimento através de seu SDK robusto, permite testes instantâneos em dispositivos reais via Expo Go, e possibilita execução web para demonstrações e apresentações, tornando o ciclo de desenvolvimento mais ágil e eficiente.
